# 小红书购买意向分析与商品推荐系统设计文档

## 1. 系统概述 (System Overview)

本系统旨在利用大语言模型 (LLM) 结合小红书 MCP (Model Context Protocol) 服务的综合能力，构建一个针对用户购买意向的深度分析与商品推荐系统。系统将通过真实的用户反馈（笔记与评论）来多维度评估商品，最终为用户提供具有说服力的、客观的商品选购建议。

## 2. 核心系统架构 (System Architecture)

整个系统分为以下几个核心层级：

### 2.1 用户交互接口层 (User Interface Layer)

- **功能**: 接收用户的自然语言购买需求，例如：“我想买一款适合大学生的轻薄本，预算5000左右，主要用来写代码和偶尔看剧”。
- **展示功能**: 负责将系统最终计算生成的商品榜单、推荐指数、多维度评价（优缺点、性价比等）进行可视化或结构化展示。

### 2.2 数据获取与检索层 (Data Acquisition Layer) - **核心依赖 MCP**

根据解析出的商品关键词，利用小红书 MCP 工具获取真实市场反馈：

- **`mcp_xhs-mcp_search_feeds`**:
  - 输入：提取出的关键词（如“5000元 轻薄本 推荐”）。
  - 输出：相关的高热度笔记列表（提取 `feed_id` 和 `xsec_token`）。
- **`mcp_xhs-mcp_get_feed_detail`**:
  - 输入：`feed_id` 和 `xsec_token`，开启 `load_all_comments=true`。
  - 输出：获取到头部热门笔记的正文内容（包含博主测评）以及海量用户评论（用于提取真实优缺点和好差评率）。

### 2.3 大模型深度分析层 (LLM Analysis Layer)

利用 LLM 的篇章理解和情感分析能力处理海量非结构化文本：

- **博文图文分析**: 从笔记正文中提取商品的**客观配置**、**核心卖点**、**博主主观评价**。
- **评论情感分析 (Sentiment Analysis)**:
  - 遍历下方的真实评论，识别“种草（好评）”和“拔草（差评）”言论。
  - 汇总高频词汇并总结为具体的**优点**和**缺点**。
  - 计算 **好评度**（正面情绪评论占比）和 **差评度**（负面及踩雷评论占比）。

### 2.4 推荐与评分引擎 (Recommendation & Scoring Engine)

设计一个多维度的综合打分公式：

- **性价比分 (0-10):** 结合博文提到的价格与硬件参数/体验指标评估。
- **好评度 (Percentage):** `(正面评论数 / 有效情感评论总数) * 100%`。
- **差评度 (Percentage):** `(负面评论数 / 有效情感评论总数) * 100%`。
- **推荐指数 (0-100):** `综合得分 = (权重A * 文本相关性) + (权重B * 性价比) + (权重C * 好评度) - (权重D * 差评度)`

## 3. 核心工作流设计 (Workflow Design)

**Step 1: 意图识别与多轮澄清 (Intent Parsing & Interactive Clarification)**

- 利用状态机设计固定追问逻辑，由 LLM 作为实体提取器处理用户输入，提取**商品类目**、**预算范围**、**核心需求**。
- 若缺少任一核心要素，则依据代码预设提示语（免受 LLM 偶尔发散的干扰），自动向用户发起多轮追问。
- 收集完整后合成确定的检索关键字送往 MCP。

**Step 2: 执行小红书检索 (Search Phase)**

- 引擎调用小红书 MCP 的 `search_feeds` 接口，传入关键字搜索笔记。
- 提取排名前 N 篇关联笔记的 `feed_id`。

**Step 3: 深度信息抓取 (Detail Fetching)**

- 异步并发调用 `get_feed_detail`，抓取 N 篇笔记的标题、正文及最多 M 条（例如：50-100条）关键评论数据。

**Step 4: 数据清洗与信息聚合 (Data Aggregation)**

- 对不同笔记中提及的同一款商品进行**实体对齐 (Entity Resolution)**。
- 汇总该商品在多篇笔记下的所有评价集。

**Step 5: 维度评估与生成 (Dimensional Evaluation)**
触发大模型分析 Prompt，令大模型输出结构化 JSON，格式要求包含：

- `product_name`: 商品名称
- `recommendation_index`: 推荐指数（90/100）
- `pros`: 优点列表（数组）
- `cons`: 缺点列表（数组）
- `cost_performance`: 性价比得分
- `positive_rate`: 好评度百分比
- `negative_rate`: 差评度百分比

**Step 6: 排序与输出呈现 (Ordering & Output)**

- 系统根据 `recommendation_index` 大小对所有的候选商品降序排列。
- 格式化输出最终多维对比表格或卡片给用户。

## 4. 关键 Prompts 设计思路 (Prompt Engineering)

在**大模型深度分析层**，我们需要向 LLM 发送如下类似 Prompt 以保证准确提取：

> “你是一个专业的商品测评数据分析师。以下是多篇关于某款产品的测评文章及评论区文本。
> 请执行情感分析和内容提炼，最终输出此产品的评价报告：
>
> 1. 根据评论用户的褒贬，统计估算好评度与差评度。
> 2. 结合价格和体验汇总其性价比得分（十分制）。
> 3. 总结前3大真实优点和前3大不可忽视的缺点。
> 文本内容：[插入 `get_feed_detail` 获取的内容]”

## 5. 潜在挑战与解决方案 (Challenges & Mitigations)

1. **反爬与限流保护**:
   - *方案*: 避免超高并发调用 MCP。适度使用 `limit` 和 `reply_limit` 参数，仅抓取前排高价值评论。
2. **水军过滤**:
   - *方案*: 提示 LLM 在分析时过滤掉过于短小、同质化的"刷榜"评论（例如只写“好看”、“赞”），侧重提取包含具体功能痛点的有效长评论。
3. **Cookie 过期**:
   - *方案*: 在底层捕捉状态码。若调用 MCP 返回无效 Token 或未登录，自动触发 `get_login_qrcode` 获取二维码提示用户扫码，或走系统持久化续期逻辑。
