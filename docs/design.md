# 小红书购买意向分析与商品推荐系统设计文档

## 1. 系统概述 (System Overview)

本系统旨在利用大语言模型 (LLM) 的**联网搜索能力**与小红书 MCP (Model Context Protocol) 服务的综合能力，构建一个针对用户购买意向的深度分析与商品推荐系统。系统采用**"全网筛选 + 小红书验证"的双阶段架构**：首先通过 LLM 联网搜索全网商品信息（销量排行、专业评测、性价比对比），筛选出最佳候选商品；然后通过小红书 MCP 获取真实用户反馈进行二次验证，最终为用户提供具有说服力的、客观的商品选购建议。

## 2. 核心系统架构 (System Architecture)

整个系统分为以下几个核心层级：

### 2.1 用户交互接口层 (User Interface Layer)

本系统提供两种交互模式：

#### Web 界面模式（主推）

基于 **Vue 3 + WebSocket** 的全双工实时通讯界面，由 `frontend/` 目录维护。

- **输入功能**: 左侧对话侧栏（`Sidebar.vue`）接收用户自然语言需求，支持多轮追问回答。
- **进度可视化**: 后端每推送一条 `status` 消息，前端即更新「分析进度卡片」（含旋转图标与进度条），阶段进度映射如下：

  | 后端 `stage` | 进度条 |
  |-------------|--------|
  | Phase 0 | 15% |
  | Phase 1 | 40% |
  | Phase 2 | 65% |
  | Phase 3 | 85% |

- **需求确认卡片**: 收到 `intent` 消息后，以绿色结构化卡片展示「目标商品 / 预算范围 / 核心诉求 / 生成搜索词」，替代原来的纯文本系统消息。
- **结果展示**: 分析完成后渲染 `ProductCard` 商品推荐卡片列表，展示推荐指数、好差评率、性价比及优缺点。

**WebSocket 消息协议（后端 → 前端）：**

| `type` | 关键字段 | 前端行为 |
|--------|----------|----------|
| `status` | `stage`, `message` | 更新分析进度卡片，同步步骤指示器 |
| `question` | `question`, `options?` | 展示追问气泡，进入等待回答状态 |
| `intent` | `data` (category/budget/core_needs) | 展示需求确认卡片，清除等待状态 |
| `completed` | `final_report` | 移除进度卡片，渲染商品推荐榜单 |
| `error` | `message` | 移除进度卡片，展示错误提示 |
| `pipeline_end` | — | 移除进度卡片，终止等待状态 |
| `info` / `warning` | `message` | 展示系统提示消息 |

**WebSocket 消息协议（前端 → 后端）：**

| `type` | `content` | 说明 |
|--------|-----------|------|
| `query` | 用户输入文本 | 首次提问 |
| `answer` | 用户追问回答 | 多轮对话中回答系统追问 |

#### 终端 CLI 模式

通过 `main.py` 运行的命令行交互程序，适合调试与无界面环境。接收用户自然语言输入，将最终商品推荐榜单以格式化文本输出到终端，同步写入 `output/` 目录下的 Markdown 报告文件。

### 2.2 LLM 联网搜索层 (Web Search Layer) - **Phase 1 核心**

利用 LLM 的联网搜索能力（通过阿里云百炼 `enable_search` 参数），从全网检索符合用户需求的商品信息：

- **电商销量排行榜**: 京东、天猫、拼多多等主流平台的热销排名
- **专业产品评测**: 什么值得买、中关村在线、ZEALER 等科技媒体的深度评测
- **性价比分析**: 同价位段不同品牌/型号的横向对比
- **市场口碑概况**: 主流渠道的综合好评率和用户反馈趋势

**技术实现**: 使用 DashScope OpenAI 兼容接口的 `extra_body={"enable_search": True}` 参数，让 Qwen 模型在回答时自动检索互联网信息。
参考文档: [阿里云百炼联网搜索](https://help.aliyun.com/zh/model-studio/qwen-api-via-openai-chat-completions)

### 2.3 数据获取与检索层 (Data Acquisition Layer) - **Phase 2 核心依赖 MCP**

根据 Phase 1 筛选出的候选商品，利用小红书 MCP 工具获取**真实用户反馈**进行二次验证：

- **`mcp_xhs-mcp_search_feeds`**:
  - 输入：针对每个候选商品生成的精准搜索关键词。
  - 输出：相关的高热度笔记列表（提取 `feed_id` 和 `xsec_token`）。
- **`mcp_xhs-mcp_get_feed_detail`**:
  - 输入：`feed_id` 和 `xsec_token`，开启 `load_all_comments=true`。
  - 输出：获取到头部热门笔记的正文内容（包含博主测评）以及海量用户评论（用于提取真实优缺点和好差评率）。

### 2.4 大模型深度分析层 (LLM Analysis Layer) - **Phase 3 综合打分**

综合全网搜索信息和小红书真实用户反馈，利用 LLM 进行最终的多维度评估：

- **全网信息作为"硬实力基准"**: 参数配置、价格竞争力、专业评测成绩
- **小红书反馈作为"软实力验证"**: 真实使用体验、用户满意度、隐藏槽点
- **评论情感分析 (Sentiment Analysis)**:
  - 从小红书评论中识别"种草（好评）"和"拔草（差评）"言论
  - 汇总高频词汇并总结为具体的**优点**和**缺点**
  - 计算 **好评度**（正面情绪评论占比）和 **差评度**（负面及踩雷评论占比）

### 2.5 推荐与评分引擎 (Recommendation & Scoring Engine)

设计一个多维度的综合打分公式：

- **性价比分 (0-10):** 结合全网评测的价格与参数对比 + 小红书用户实际体验反馈。
- **好评度 (Percentage):** `(正面评论数 / 有效情感评论总数) * 100%`。
- **差评度 (Percentage):** `(负面评论数 / 有效情感评论总数) * 100%`。
- **推荐指数 (0-100):** `综合得分 = (权重A * 全网评测得分) + (权重B * 性价比) + (权重C * 小红书好评度) - (权重D * 差评度)`

**Step 1: 意图识别与多轮澄清 (Intent Parsing & Interactive Clarification)**

- 利用状态机设计固定追问逻辑，由 LLM 作为实体提取器处理用户输入，提取**商品类目**、**预算范围**、**核心需求**。
- 若缺少任一核心要素，则依据代码预设提示语（免受 LLM 偶尔发散的干扰），自动向用户发起多轮追问。
- 收集完整后合成确定的检索关键字。

**Step 2: LLM 联网全网搜索 (Phase 1: Web Search)**

- 调用 DashScope OpenAI 兼容接口，通过 `extra_body={"enable_search": True}` 启用联网搜索。
- LLM 自动检索全网信息，包括电商销量排行榜、专业评测、性价比对比等。
- 输出结构化的 `WebSearchReport`，包含市场概况和 3~5 款候选商品。
- 每个候选商品包含精准的小红书搜索关键词，用于下一阶段的二次验证。

**Step 3: 小红书二次搜索验证 (Phase 2: XHS Verification)**

- 针对 Phase 1 筛选出的每款候选商品，使用其专属关键词在小红书上进行精准搜索。
- 调用 MCP 的 `search_feeds` + `get_feed_detail` 获取真实用户笔记和评论。
- 每款商品从小红书取前 3 篇相关笔记的详情和评论。

**Step 4: 综合深度分析 (Phase 3: Comprehensive Analysis)**

- 将全网搜索信息（硬实力基准）与小红书真实反馈（软实力验证）合并提交给 LLM。
- LLM 进行情感分析、优缺点提炼、多维度打分。
- 输出最终的 `RecommendationReport`，包含结构化的推荐报告卡片。

**Step 5: 排序与输出呈现 (Ordering & Output)**

- 系统根据 `recommendation_index` 大小对所有的候选商品降序排列。
- 格式化输出最终多维对比表格或卡片给用户。

## 4. 数据模型设计 (Data Models)

### 4.1 Phase 1 输出: WebSearchReport

```python
class CandidateProduct(BaseModel):
    product_name: str       # 具体商品名称/型号
    brand: str              # 品牌名称
    price_range: str        # 参考价格区间
    highlights: List[str]   # 核心卖点（不超过5条）
    sales_info: str         # 销量/市场热度信息
    search_keyword_for_xhs: str  # 用于小红书搜索的精准关键词

class WebSearchReport(BaseModel):
    market_summary: str             # 市场概况简述
    candidates: List[CandidateProduct]  # 3~5款候选商品
```

### 4.2 Phase 3 输出: RecommendationReport

```python
class ProductEvaluation(BaseModel):
    product_name: str           # 商品名称
    recommendation_index: int   # 推荐指数(0-100)
    pros: List[str]            # 优点列表
    cons: List[str]            # 缺点列表
    cost_performance: float    # 性价比(0-10)
    positive_rate: str         # 好评度
    negative_rate: str         # 差评度
    summary: str               # 总结性购买建议

class RecommendationReport(BaseModel):
    recommendations: List[ProductEvaluation]  # 按推荐指数降序排列
```

## 5. 关键 Prompts 设计思路 (Prompt Engineering)

### Phase 1 联网搜索 Prompt
>
> "你是一个专业的商品调研分析师。请联网搜索当前市场上符合用户需求（类目/预算/偏好）的商品
> 重点参考：1.电商销量排行 2.专业评测 3.性价比分析 4.用户口碑概况
> 筛选出3~5款最值得推荐的候选商品，并为每款商品生成适合小红书搜索的精准关键词。"

### Phase 3 综合分析 Prompt
>
> "你是一个精通全网商品测评的数据分析师。
> 下面是两部分数据：
> 1.【全网搜索结果】：来自各大电商平台、专业评测网站的综合信息（硬实力基准）
> 2.【小红书真实反馈】：来自小红书的真实用户笔记和评论（软实力验证）
> 请综合两部分信息，忽略水军评论，输出最终的多维度推荐报告。"

## 6. 潜在挑战与解决方案 (Challenges & Mitigations)

1. **LLM 联网搜索信息时效性**:
   - *方案*: 使用 DashScope 的 `enable_search` 可自动获取最新互联网信息，确保候选商品为当前在售型号。
2. **反爬与限流保护**:
   - *方案*: 避免超高并发调用 MCP。适度使用 `limit` 和 `reply_limit` 参数，仅抓取前排高价值评论。
3. **水军过滤**:
   - *方案*: 提示 LLM 在分析时过滤掉过于短小、同质化的"刷榜"评论（例如只写"好看"、"赞"），侧重提取包含具体功能痛点的有效长评论。
4. **Cookie 过期**:
   - *方案*: 在底层捕捉状态码。若调用 MCP 返回无效 Token 或未登录，自动触发 `get_login_qrcode` 获取二维码提示用户扫码，或走系统持久化续期逻辑。
5. **候选商品与小红书内容匹配度**:
   - *方案*: Phase 1 输出的 `search_keyword_for_xhs` 需要足够精准（包含品牌+型号），避免在小红书上搜到无关内容。若某候选商品在小红书上无足够资料，在最终报告中标注"真实用户反馈不足"。
